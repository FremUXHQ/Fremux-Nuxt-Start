#!/bin/bash
# 🚀 **POP018: Estratégia de Repositório para Projetos Públicos e Privados com Submódulos Git**  

## 📌 Objetivo  
Este POP orienta como configurar um repositório público contendo um submódulo privado, garantindo acesso controlado.  

---
## 🏗️ 1️⃣ Testar se o Git está configurado corretamente
```bash
git config --global user.name && git config --global user.email
```
✔ Se o nome e o email forem exibidos, o Git está configurado corretamente.  

🔹 **Testar se a chave SSH está funcional**  
```bash
ssh -T git@github.com
```
✔ Se a resposta for `"Hi <usuario>! You've successfully authenticated, but GitHub does not provide shell access."`, o SSH está funcionando.  

🔹 **Testar autenticação via GitHub CLI (`gh`)**  
```bash
gh auth status
```
✔ Se `Logged in as <usuario>` aparecer, a autenticação foi feita corretamente.  

⚠ **Se qualquer um desses testes falhar, revise a execução do [POP017](./POP017.md).**  

---

## 🏗️ 2️⃣ Verificar se o repositório será pessoal ou de organização**  

### 📌 **Identificar o tipo de repositório antes da listagem**  
Antes de listar os repositórios, precisamos determinar se estamos lidando com **repositórios pessoais ou de uma organização**.  

🔹 **Caso seja um repositório pessoal**, use:  
```bash
gh repo list --limit 50
```
✔ Esse comando listará **todos os repositórios do seu perfil pessoal**.  

🔹 **Caso seja um repositório dentro de uma organização**, substitua `<organizacao>` pelo nome correto e use:  
```bash
gh repo list <organizacao> --limit 50
```
✔ Esse comando listará **todos os repositórios pertencentes à organização informada**.  

### 📌 **Definição do formato dos links Git**  
📌 **Agora que identificamos o contexto, todos os comandos seguintes devem seguir o formato correto:**  

- **Se for um repositório pessoal, use:**  
```bash
git@github.com:usuario/repo.git
```
- **Se for um repositório dentro de uma organização, use:**  
```bash
git@github.com:organizacao/repo.git
```
⚠️ **Atenção:** **De agora em diante, sempre que aparecer `usuario` (como em `usuario/repo.git`), substitua por  `organizacao` (como em `organizacao/repo.git`) se estiver em um contexto organizacional.**

---
## 🏗️ 3️⃣ Verificar se os repositórios já existem**  

📌 **Testar se o repositório público já existe:**  
```bash
gh repo view usuario/public_repo
```
✔ Se ele já existir, o comando mostrará detalhes sobre ele.  

📌 **Testar se o repositório privado já existe:**  
```bash
gh repo view usuario/private_repo
```
✔ Se ele já existir, podemos vinculá-lo ao repositório público.  

⚠ **Se um dos repositórios não existir, siga para a seção [4 - Criando os Repositórios](#️-4️⃣-criando-os-repositórios-caso-ainda-não-existam).**  


## 🏗️ 4️⃣ Criando os repositórios caso ainda não existam**

Caso os repositórios ainda não foram criados (confirme na seção [3 - Verificar se os repositórios já existem](#️-3️⃣-verificar-se-os-repositórios-já-existem)) execute:

```bash
gh repo create public_repo --public --confirm --add-readme
gh repo create private_repo --private --confirm --add-readme
```
Se for uma **organização**:  
```bash
gh repo create organizacao/public_repo --public --confirm --add-readme
gh repo create organizacao/private_repo --private --confirm --add-readme
```

## 🏗️ 5️⃣ Repositório público e submódulo**  

O padrão deste POP assume que teremos um repositório principal (público ou provado - aqui no exemplo é público) e que, dentro dele, exista um repositório privado (sempre) que, adicionado na forma de um `submódulo`, será o local onde partes ainda em estudo, arquivos adicionais, documentações internas e arquivos privados ao repositório oficial serão incluidos.

## 📌 **5.1 - Entendendo o Arquivo `.gitmodules` no GitHub**  

O Git armazena informações sobre submódulos dentro do arquivo `.gitmodules`. Esse arquivo contém a estrutura dos submódulos vinculados ao repositório e é essencial para que outros usuários e ambientes possam atualizar ou clonar corretamente os submódulos.  

### 🔍 **O que o `.gitmodules` contém?**  
O arquivo `.gitmodules` é um arquivo de configuração que define quais submódulos existem no repositório e onde eles estão localizados. Exemplo:  

```ini
[submodule "private"]
	path = private
	url = git@github.com:usuario/private_repo.git
```

✔ **`submodule "private"`** – Nome do submódulo  
✔ **`path = private`** – Caminho onde o submódulo está localizado dentro do repositório  
✔ **`url = git@github.com:usuario/private_repo.git`** – Repositório remoto vinculado  

Se um repositório contém submódulos, o arquivo `.gitmodules` deve estar presente no GitHub e pode ser acessado via **API do GitHub**.

## 🏗️ 6️⃣ Garantir que o repositório está clonado**

📌 **1 - Verificando**  

Se o repositório já estiver clonado, entre no diretório dele e valide se o repositório está corretamente configurado:  

```bash
git status
```
✔ Se o repositório já foi clonado corretamente, o comando `git status` deve exibir algo como:  
```
On branch main
Your branch is up to date with 'origin/main'.
nothing to commit, working tree clean
```
---

📌 **2 - Clonando o Repositório**

Caso o repositório ainda **não tenha sido clonado**, execute:  
```bash
git clone --recurse-submodules git@github.com:usuario/public_repo.git
cd public_repo
```
📌 **Se o repositório for de uma organização, substitua `usuario` pelo nome da organização:**  
```bash
git clone --recurse-submodules git@github.com:organizacao/public_repo.git
cd public_repo
```
⚠ **Esse comando já clona o repositório e seus submódulos (se houverem). Se não houver submódulos, a próxima etapa será adicioná-los manualmente.**  

Agora, siga para a próxima seção para verificar se o submódulo já está presente.  

---
## 🏗️ 7️⃣ Verificar se o repositório público já tem um submódulo**  

Agora que o repositório está clonado corretamente, precisamos verificar se ele já contém um submódulo vinculado.  

### 📌 **Opção 1: Se o repositório já foi clonado, verificar diretamente os submódulos**
Agora que estamos dentro do repositório público, execute: 

```bash
git submodule status
```
✔ **Se esse comando retornar um hash e um caminho, significa que o submódulo já foi adicionado corretamente.**  

⚠ **Se o comando não retornar nada, significa que o repositório ainda não tem submódulos. Siga para a seção [8 - Adicionando o Submódulo](#️-8️⃣-adicionando-o-submódulo).**  

---
### 📌 **Opção 2: Se o repositório ainda não foi clonado, verificar os submódulos via API do GitHub (`gh`)**  
Caso queira verificar remotamente, sem clonar o repositório, podemos acessar a API do GitHub:  

- **Para repositórios pessoais, use:**  
```bash
gh api repos/usuario/public_repo/contents/.gitmodules --jq 
```

✅ **Se esse comando retornar um conteúdo válido, significa que o submódulo já está configurado no repositório.**  

⚠ **Se o comando não retornar nada ou exibir `"Not Found"`, significa que o submódulo ainda não foi adicionado. Siga para a seção [8 - Adicionando o Submódulo](#️-8️⃣-adicionando-o-submódulo).**  

## 🏗️ 8️⃣ Adicionando o Submódulo

Adicione o submódulo ao repositório público e faça o commit:  

```bash
git submodule add git@github.com:usuario/private_repo.git private/
git submodule update --init --recursive
git add .gitmodules private/
git commit -m "Adiciona submódulo privado"
git push origin main
```

📌 **Se o repositório for de uma organização, substitua `usuario` pelo nome da organização:**  

```bash
git submodule add git@github.com:organizacao/private_repo.git private/
```

---
### 📌 Verificar novamente se o submódulo foi adicionado corretamente**
Agora que o submódulo foi adicionado, precisamos garantir que ele está configurado corretamente.  
Siga para a seção **[7 - Verificar se o Repositório já tem um Submódulo](#️-7️⃣-verificar-se-o-repositório-público-já-tem-um-submódulo)** e valide a configuração.
